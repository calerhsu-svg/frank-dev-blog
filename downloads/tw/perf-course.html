第一章：效能分析工具入門

效能優化的第一步是「量測」，而非猜測。Unity Profiler 和 Cocos 的 Debug 面板是你最重要的武器。

1.1 Unity Profiler

開啟 Profiler：Window → Analysis → Profiler (Ctrl+7)

關鍵指標：
├── CPU Usage    → 每幀 CPU 耗時
├── GPU Usage    → 渲染耗時
├── Memory       → 記憶體分配
├── Rendering    → Draw Call、三角面數
└── Physics      → 物理計算耗時

目標值（60fps）：
  每幀預算 = 1000ms / 60 = 16.67ms
  CPU < 12ms（留餘裕給 GPU）
  Draw Call < 100（手機）/ < 500（PC）
  三角面 < 100K（手機）
💡 永遠在目標設備上測試，Editor 中的效能數據不代表實機表現。

1.2 Cocos Creator Debug

// 開啟效能面板
cc.debug.setDisplayStats(true);

// 關鍵指標
FPS        → 幀率
Draw Call  → 渲染批次
Triangles  → 三角面數量
Sprites    → 精靈數量

第二章：Draw Call 優化與批次合併

Draw Call 是 CPU 向 GPU 發出的繪製指令。每次 Draw Call 都有 CPU 開銷，減少 Draw Call 是最有效的優化手段之一。

2.1 圖集 (Sprite Atlas)

// Unity：建立 Sprite Atlas

// 將所有同場景的 Sprite 拖入 Atlas

// 效果：
// 優化前：每個 Sprite 一個 Draw Call → 50 Draw Calls
// 優化後：整個 Atlas 一個 Draw Call → 1 Draw Call

2.2 動態合批與靜態合批

// Unity 靜態合批（Static Batching）
// Inspector → Static 勾選 ✅
// 適用：不會移動的場景物件（地形、建築、裝飾）

// Unity 動態合批（Dynamic Batching）

// 限制：頂點數 < 300、相同材質

// SRP Batcher（推薦）

⚠️ 不同材質、不同 Shader、不同 Texture 的物件無法合批。統一材質是關鍵。

第三章：記憶體管理與資源載入策略

3.1 資源生命週期

// 延遲載入（Lazy Loading）
Resources.LoadAsync<Sprite>("path", (sprite) => {
    renderer.sprite = sprite;
});

// Addressables 系統（推薦）
var handle = Addressables.LoadAssetAsync<GameObject>("enemy");
handle.Completed += (op) => {
    Instantiate(op.Result);
};

// 卸載未使用資源
Resources.UnloadUnusedAssets();
System.GC.Collect();

3.2 Texture 壓縮

平台推薦壓縮格式：
├── Android → ETC2 / ASTC
├── iOS    → ASTC（最佳壓縮比）
├── PC     → DXT5 / BC7
└── WebGL  → ETC2 + fallback

技巧：
  - 尺寸用 2 的冪次（256, 512, 1024）
  - UI 圖不需要 Mipmap
  - 降低 Max Size 到實際需要的大小

第四章：物理引擎效能調校

// Unity Physics 設定
// Edit → Project Settings → Physics 2D

// 關鍵調整：
Fixed Timestep = 0.02 (50Hz)  // 不要低於 0.01
Velocity Iterations = 6       // 降低可提升效能
Position Iterations = 2
Layer Collision Matrix → 取消不需要的碰撞層

// 使用簡單的碰撞形狀
✅ BoxCollider2D, CircleCollider2D
❌ PolygonCollider2D（頂點多，計算量大）

// 物件休眠
Rigidbody2D.sleepMode = RigidbodySleepMode2D.StartAsleep;
💡 Layer Collision Matrix 是最容易被忽略的優化。關閉不需要的碰撞層可以大幅減少物理計算。

6.1 減少 GC 分配

    string text = "Score: " + score;  // 每幀 new string
    List<Enemy> list = new List<Enemy>();  // 每幀 new List

    scoreText.text = sb.ToString();

    reusableList.Clear();
    // 填充 reusableList...
}

6.2 快取 Component 參照

// ❌ 每幀 GetComponent
void Update() {
    GetComponent<Rigidbody2D>().velocity = v;
}

// ✅ 快取
Rigidbody2D rb;
void Awake() { rb = GetComponent<Rigidbody2D>(); }
void Update() { rb.velocity = v; }

第八章：實戰案例：從 30fps 到 60fps

以一款 2D 動作遊戲為例，我們在 Android 中端機上將幀率從 30fps 提升到穩定 60fps。

優化步驟

Step 1：Profiler 分析
  → CPU 瓶頸：UI Canvas 重建佔 8ms
  → Draw Call：287（太高）
  → GC Alloc：每幀 2.4KB

Step 2：Draw Call 優化
  → Sprite Atlas 合併 → 287 降到 42
  → 節省 6ms CPU 時間

Step 3：UI Canvas 分層
  → 靜態/動態分離 → UI 重建從 8ms 降到 1ms

Step 4：GC 優化
  → 字串快取 + 物件池 → GC 從 2.4KB/幀降到 0

Step 5：物理優化
  → Layer Matrix + 簡化碰撞器 → 物理從 3ms 降到 0.8ms

結果：總幀時間從 33ms → 14ms（穩定 60fps）
